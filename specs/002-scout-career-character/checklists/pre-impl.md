[Consistency, Spec §FR-001, Plan §Complexity Tracking] — **Fixed 2026-06-18**: FR-001 updated to state that adding optional params with backward-compatible defaults does not violate "zero algorithm changes" because the existing code path is fully preserved when all new params take their defaults, and the per-career term-processing logic is unreachable by those params.

[Clarity, Spec §FR-008] — **Fixed 2026-06-18**: FR-008 updated to specify that `bypass_qualification=True` skips the **entire** enlistment check: both the DM computation (`qual_dm = _dm(...)`) and the 2D6 roll+comparison (`roller.roll(6, count=2) + qual_dm < career.qualification_target`). No partial skip. Also added matching precision to plan.md Complexity Tracking.
- [x] CHK058 - Is the `preset_characteristics` caller contract specified: if supplied with a characteristic value below the career's qualification threshold, does `generate_character` validate and reject, or does it trust the caller to have pre-validated before calling? [Clarity, Completeness, Spec §FR-008] — **Fixed 2026-06-18**: FR-008 updated to specify that `generate_character` trusts `preset_characteristics` without re-validation; `roll_until_qualified` is the sole authority for guaranteeing the threshold is met. ## Edge Case & Exception Flow Coverage

- [x] CHK059 - Is it specified whether the qualification re-roll loop is purely infinite (bounded only by system resources), or whether there is any halt condition beyond the qualification threshold being met? [Completeness, Edge Case, Spec §FR-008] — **PASS 2026-06-18**: Already specified. The Edge Cases section states "no maximum re-roll limit is imposed, consistent with the SRD intent of guaranteed enrollment when a career is chosen." FR-008 confirms the loop runs until the threshold is met with no other halt condition. No spec change needed.
- [x] CHK060 - Is the behaviour specified for `--career` when the input normalises to an empty string (e.g., `--career "   "` strips to `""`)? Should it be treated as a missing flag (→ draft) or as an unrecognised career (→ exit 1 + error)? [Edge Case, Clarity, Spec §FR-007] — **Fixed 2026-06-18**: Added clarification to spec.md Session 2026-06-18. Whitespace-only input normalises to `""`, which is not in `CAREER_REGISTRY`; FR-012 applies (exit 1, stderr `Unknown career '   '. Valid careers: navy, scout` with original pre-normalization input preserved).
- [x] CHK061 - Is the behaviour specified when `generate_career_character` is passed a `Career` whose `qualification_target` is 0 (i.e., always qualifies on the first roll), to confirm the loop terminates immediately rather than requiring special-casing? [Edge Case, Completeness] — **PASS 2026-06-18**: Logical consequence of the loop condition `characteristics[career.qualification_stat] >= career.qualification_target`. Any characteristic value satisfies `>= 0` on the first roll; the loop exits immediately. No special-casing is needed. No spec change required.
- [x] CHK062 - Is the behaviour specified when `roll_until_qualified` receives a `Career` whose `qualification_stat` is `None` (draft-assigned careers bypass the loop entirely, but could this function be called with such a career)? [Edge Case, Completeness, Spec §FR-008] — **Fixed 2026-06-18**: Added clarification to spec.md Session 2026-06-18. `qualification_stat=None` is a caller precondition violation; behavior is undefined and no defensive handling is required. All `CAREER_REGISTRY` entries must supply a non-None `qualification_stat`; both current careers (Navy, Scout) satisfy this.
